# Project Instructions & Rules

These rules apply to all changes, additions, and updates made to this repository.

## 1. Professional Showcase & Avoiding "Recruitment Fishing"
- **Focus**: The website should showcase deep technical expertise, architecture decisions, technology leadership, and engineering ROI.
- **Aesthetic/Tone**: It should maintain a premium, state-of-the-art developer/architect portfolio feel rather than looking like a generic template for recruiters.
- **Content**: Highlight actual case studies, architectural matrixes, and strategic reasoning rather than buzzword-heavy recruitment traps.

## 2. Complete Anonymity & Privacy
- **Rule**: Do NOT use or reference the user's current or previous employers' company names under any circumstances.
- **Terminology**: Use generalized descriptions (e.g., "global enterprise SaaS infrastructure," "managed services operations," "leading security services integration") instead of actual company titles.

## 3. Branching & Git Strategy
- **Rule**: All development, testing, and modifications must be performed on the `qa` branch.
- **Verification**: Changes should be reviewed and verified on the `qa` branch before merging or pushing to the `main` branch.
